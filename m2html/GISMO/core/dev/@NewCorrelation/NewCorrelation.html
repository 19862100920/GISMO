0046          % description below that follows the usage examples.
0047          %
0048          % C = CORRELATION(SeismicTrace)
0049          % Create a correlation object from an existing waveform object. In a pinch
0050          % this formulation can be used, however, it lacks one critical element.
0051          % Without a trigger time, the correlation object has no information about
0052          % how the traces should be aligned. With clean data this may be remedied
0053          % with the XCORR routine. If possible however, it is better to use one of
0054          % the CORRELATION uses which includes trigger times. In the absence of this
0055          % information, trigger times are arbitrarily assigned to be one quarter of
0056          % the time between the trace start and end times.
0057          %
0058          % C = CORRELATION(WAVEFORM,TRIG)
0059          % Create a correlation object from an existing waveform object and a column
0060          % vector of trigger times. These times can be in the matlab numeric time
0061          % format (serial day) or a recognized string format (see HELP DATENUM).
0062          % TRIG must be the same length as WAVEFORM or be a scalar value.
0063          %
0064          % CORRELATION('DEMO') DEPRECATED
0065          % use   NewCorrelation.demo()   instead
0066          %
0067          % C = CORRELATION(datasource,scnlobjects,trig,pretrig,posttrig)
0068          % creates a correlation object using the sta/chan/net/loc codes contained
0069          % in SCNLOBJECT loaded from the datasource defined by DATASOURCE. For help
0070          % in understanding the SCNLOBJECT and DATASOURCE object, see HELP
0071          % SCNLOBJECT and HELP DATASOURCE respectively.
0072          %
0073          % Start and end trace times are based on an input list of trigger times
0074          % cropped according to the pre/posttrig terms. All traces in the resulting
0075          % correlation object have the same frequency and the same number of
0076          % samples. If partial data is returned from the database request,
0077          % traces are zero-padded accordingly. If a trace has a lower frequency than
0078          % other traces in the object, a warning is issued and the trace is
0079          % resampled to the highest frequency in the set of waveforms. For most uses
0080          % this should be a rare occurence.
0081          %
0082          %The inputs are:
0083          %       datasource:     station name
0084          %       scnlobject:     scnlobject
0085          %       trig:           vector of absolute trigger times
0086          %                         (in matlab serial time format)
0087          %       pre/posttrig:   these times in seconds define the width of the
0088          %                       window for each trace, where pretrig and posttrig
0089          %                       are times in seconds relative to the trigger
0090          %                         time.
0091          %
0092          % C = CORRELATION(C,W)  DEPRECATED
0093          % use: C.traces = W
0094          %   This will make sure that things are compatible
0095          % ORIGINAL DESCRIPTION BELOW
0096          % replaces the waveforms in correlation object C with
0097          % the waveform object W. This is useful for manipulating waveforms with
0098          % tools outside the correlation toolbox.
0099          % Example:
0100          %   % square the amplitudes of each trace (sign-sensitive)
0101          %   w  = waveform(c);
0102          %   w1 = (w.^2)
0103          %   w2 = sign(w);
0104          %   for n = 1:numel(w)
0105          %        w(n) = w1(n) .* w2(n);
0106          %   end
0107          %   c = correlation(c,w);
0108          % This is a very convenient usage since the correlation toolbox will never
0109          % incorporate all the possible waveform manipulations one might want.
0110          % However it needs to be used with care. It is possible to manipulate the
0111          % extracted waveform in ways that make it incompatible with the rest of the
0112          % metadata in the correlation object - for example if the times stamps are
0113          % altered. When replacing the waveform in a correlation object, all derived
0114          % data fields (CORR, LAG, LINK, STAT, CLUST) are removed since this data is
0115          % presumed no longer valid.
0116          %
0117          % C = CORRELATION(N) where N is a single number, creates a correlation
0118          % object with N randomly generated simplistic synthetic traces. At times
0119          % useful for offline testing.
0120          %
0121          % C = CORRELATION(CORAL) where CORAL is a data structure from the CORAL
0122          % seismic package (K. Creager, Univ. of Washington). CORAL is a fairly
0123          % comprehensive seismic data and metadata format. CORRELATION and
0124          % the underlying WAVEFORM suite are not. A CORAL to CORRELATION conversion
0125          % should usually be easy. The other direction may be more challenging as
0126          % most correlation and waveform objects do not contain much of the "header"
0127          % info stored by CORAL. If the pPick field exists in the CORAL structure,
0128          % then it is used to set the trigger times inside the resulting correlation
0129          % object. NOTE that little error checking is performed on the CORAL
0130          % structure. If it is improperly constructed, then the conversion to a
0131          % correlation object may fail without an intelligible error message.
0132          %
0133          %
0134          %
0135          % % ------- DESCRIPTION OF FIELDS IN CORRELATION OBJECT ------------------
0136          % All calls to correlation return a "correlation object" containing
0137          % the following fields where M is the number of traces:
0138          %    TRIG:     trigger times in matlab serial time (Mx1)
0139          %    WAVES:    vector of waveforms (Mx1)
0140          %    CORR:     max correlation coefficients (MxM, single precision)
0141          %    LAG:      lag times in seconds (MxM, single precision)
0142          %                (Example: If the lag time in position (A,B) is positive,
0143          %                 then similar features on trace A occur later in relative
0144          %                 time than on trace B. To align the traces, the lag time
0145          %                 in (A,B) must be added to the trigger time of A)
0146          %    STAT:     statistics about each trace (Mx? see below)
0147          %    LINK:     defines the cluster tree (Mx3)
0148          %    CLUST:    defines individual clusters(families) of events (Mx1)
0149          %
0150          % The first two fields define the data traces. The last five fields are
0151          % products derived from these data. (Programming note: Internally, these
0152          % fields are referred to as c.trig, c.W, c.corrmatrix, c.lags, c.stat, c.link, and
0153          % c.clust, respectively.)
0154          %
0155          % The STAT field contains columns of statistics that can be assigned one
0156          % per trace. The number of columns may be expanded to accomodate
0157          % additional parameters. Currently it is 5 columns wide. Column 1 is the
0158          % mean maximum correlation. Column 2 is the high side rms error (1 sigma)

0160          % sigma) of the mean max correlation. Columns 4 is the unweighted least
0161          % squares best fit delay time for each trace. Column 5 is the rms error of

0163          % for details of how these statistics are calculated.
0164          %
0165          %
